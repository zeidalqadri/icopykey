# JSON trace fixtures

Captured (or synthetic) MIFARE Classic auth traces in the JSON format
that `icopyzed crack --from-trace FILE` consumes. The canonical schema
is documented in the docstring of
`NestedAttack.from_trace_file` (in `src/icopykey/cli/crypto1_attack.py`);
this README mirrors it for quick reference and documents the field
semantics tool authors need.

## Schema

```json
{
  "uid":              "11223344",
  "target_sector":    4,
  "known_sector":     0,
  "known_key":        "ffffffffffff",
  "known_nonce":      "AABBCCDD",
  "distance_window":  [0, 320],
  "mfkey64_traces": [
    {
      "nt":     "DEADBEEF",
      "nr":     "11223344",
      "ar_enc": "55747576",
      "at_enc": "3d488c43"
    }
  ],
  "traces": [
    {
      "nt_enc":   "11223344",
      "at_enc":   "55667788",
      "nr":       "DDEEFF00",
      "distance": 160
    }
  ]
}
```

### Required fields

| Field           | Type      | Notes                                                              |
|-----------------|-----------|--------------------------------------------------------------------|
| `uid`           | 8-hex     | Card UID (4 bytes, big-endian hex).                                |
| `target_sector` | int       | Sector to recover.                                                 |

### `mfkey64_traces` (preferred — what actually recovers keys)

Each entry is one complete MIFARE Classic 4-step authentication.
Inputs are the canonical `mfkey64` arguments — plaintext `nt` and
`nr`, encrypted `ar_enc` and `at_enc`. `NestedAttack.recover_key`
runs the ported `crypto1_canonical.mfkey64` against each entry.

| Field    | Type   | Description                                                                |
|----------|--------|----------------------------------------------------------------------------|
| `nt`     | 8-hex  | Plaintext tag challenge as observed on the wire (first-auth) or recovered. |
| `nr`     | 8-hex  | Plaintext reader challenge (the reader picks this — no encryption needed). |
| `ar_enc` | 8-hex  | Encrypted reader response = `prng_successor(nt, 64) XOR ks2`.              |
| `at_enc` | 8-hex  | Encrypted tag response = `prng_successor(nt, 96) XOR ks3`.                 |

This is the **only path that recovers real keys today**. Use it
whenever your capture tool gives you a full auth trace.

### `traces` (legacy nested flow)

Older nested-attack shape, kept for backward compatibility. The
`recover_key_from_keystream` path it feeds is incomplete — see
warnings in `src/icopykey/cli/crypto1_attack.py`. Keep using this only
if you have nonces-only captures from `mfcuk -C`.

| Field      | Type   | Description                                                       |
|------------|--------|-------------------------------------------------------------------|
| `nt_enc`   | 8-hex  | Encrypted tag nonce `{nt}`.                                       |
| `at_enc`   | 8-hex  | Encrypted tag answer `{at}` — optional, used for validation.      |
| `nr`       | 8-hex  | Reader nonce `nr` (plain) — optional, used for `{at}` validation. |
| `distance` | int    | Optional PRNG-distance hint for nested-with-known-key flow.       |

### Path B (known-key nested) fields

Optional fields for the experimental nested flow:

| Field             | Type      | Description                                              |
|-------------------|-----------|----------------------------------------------------------|
| `known_sector`    | int       | Sector for which the key is known.                       |
| `known_key`       | 12-hex    | Known 6-byte key.                                        |
| `known_nonce`     | 8-hex     | Plaintext `nt` observed during the known-sector auth.    |
| `distance_window` | [int,int] | Override default PRNG-distance search window [0, 320].   |

## Converting from external tools

### mfcuk

`mfcuk -C -R <block>:<keytype>` prints `Nt = 0x...` lines for each
captured nonce. Map them into the `traces` array (one
`{"nt_enc": "..."}` per line). mfcuk doesn't expose `at_enc` or `nr`
through `-C`, so use mfcuk's own `-R` recovery (which writes keys
directly) and feed those into the local key library instead — the
`traces`-only path is currently incomplete.

For a complete auth trace (full `mfkey64_traces` entry), use a tool
that sniffs the full reader↔tag exchange (e.g., Proxmark3 `hf mf sniff`).

### Proxmark3

`hf mf sniff` captures the full auth. Extract uid, nt, nr, ar (as
sent), at (as sent) from the sniff log. ar and at come over the wire
already encrypted, so they go directly into `ar_enc` and `at_enc`.
A converter script would belong at `tools/proxmark_to_icopyzed.py`.
